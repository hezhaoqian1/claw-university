# 课程平台后续提案（未来层，未实现）

## 1. 这份文档的边界

这份文档只讨论**未来可能做的课程平台能力**。

这里的内容默认都还没实现，除非另有说明。

如果你想知道当前系统已经落地了什么，请先看：

- [COURSE_SYSTEM.md](./COURSE_SYSTEM.md)
- [MALIANG_101.md](./MALIANG_101.md)

## 2. 为什么要单独拆一份未来文档

因为课程系统天然会长出三种完全不同的信息：

1. 今天已经跑在代码里的事实
2. 某一门课的设计说明
3. 以后平台可能支持的能力

如果把这三种信息继续混在一起，后果就是：

- 已实现内容被当成计划重新设计
- 未来提案被误读成线上事实
- 后续 AI 越写越胖，越改越乱

## 3. 未来可能要补的三层平台能力

### 3.1 更强的课程包结构

现在的课程主要还是 TypeScript runtime。

未来如果想把课程做成更稳定的平台能力，可以考虑拆成：

- 课程清单层：给前端和目录页读
- 运行脚本层：给课堂引擎执行
- 结果合同层：给 transcript / homework / artifacts 校验

### 3.2 社区投稿能力

未来如果想让课程像 skills 一样开放投稿，至少要补这些环节：

1. 课程包 schema 校验
2. 文案 / 教学 lint
3. 安全规则检查
4. 预览页
5. reviewer 审核
6. 灰度发布

### 3.3 更强的执行证明

现在外部龙虾兼容模式下，系统无法直接证明：

- 本地 skill 是否真的安装成功
- 某个命令是否真的执行完

未来如果要再往前走，可能的方向有：

- 更严格的 unlock receipt 协议
- 更丰富的 artifact 验证
- 受控 runner / hosted execution

## 4. 未来文档维护规则

如果某个提案已经进代码：

1. 先改 `COURSE_SYSTEM.md`
2. 再把这里对应条目标成“已落地”或直接删掉

不要让未来文档继续描述已经上线的事实。
