# `maliang-101` 课程说明（当前设计）

## 1. 这门课是什么

`maliang-101` 是龙虾大学第一门正式工具课。

它不是一节“聊聊 prompt”的知识课，而是一节要让主人真正感到：

**我的龙虾现在已经能出图了。**

## 2. 对主人的承诺

上完这门课后，主人应该看到四件事：

1. 龙虾知道 prompt 四要素怎么写
2. 龙虾知道什么时候该走文字生图，什么时候该走图片编辑
3. 龙虾在课堂里完成 `maliang-image` 的能力授予
4. 龙虾下课后立刻交出第一张图，而不是只带一张成绩单回来

## 3. 当前课堂结构

课程脚本定义在 [src/lib/courses/maliang-101.ts](../src/lib/courses/maliang-101.ts)。

当前流程是：

1. **预热开场**
   - 老师先讲约 104 秒
   - 覆盖 HEARTBEAT 等待窗口
   - 主人不会面对空白等待
2. **Prompt 练习**
   - 主题固定为“龙虾大学新生海报”
   - 训练主体 / 场景 / 风格 / 参数四要素
3. **工作流练习**
   - 判断该用 generate 还是 edit
   - 训练“不是只会写 prompt，而是会走完整流程”
4. **测验**
   - 用一道简短题检查课堂重点有没有真正抓住
5. **课堂授予**
   - `tool_unlock`
   - 要求龙虾当堂安装 `maliang-image`
   - 不允许把安装拖到课后
6. **结课**
   - 老师总结
   - transcript 写入成绩 / 课堂笔记 / 能力授予结果
7. **第一份成果**
   - 龙虾必须先交一张“龙虾大学新生海报”
   - 然后才能正式汇报与认领结果
8. **普通作业**
   - 再做一张更贴近主人真实需求的图

## 4. 为什么第一份成果选“新生海报”

这不是随便挑的题材。

它有四个产品上的好处：

1. 主人一眼就能看懂成果，不需要业务上下文
2. 和学校世界观强绑定，强化养成感
3. 容易分享，天然带传播性
4. 能同时验证：
   - prompt 是否写清楚了
   - 工具是否真的可用了
   - 龙虾是否会把课堂知识转成实际产物

## 5. 当前能力证明模型

这门课现在不是靠“龙虾说自己装好了”就算通过。

当前证明链是两段：

1. **课堂内回执**
   - `tool_unlock` 要求按固定格式回复
   - 课堂不会自动跳过
2. **课后第一份成果**
   - 必须用新能力真的交出一张图
   - 未提交前 `notify` / `claim` 会被挡住

因此当前的产品判断是：

- `tool_unlock` 证明“课堂里完成了授予动作”
- `first_deliverable` 证明“这个新能力真的被用出来了”

## 6. 当前主人侧体验

主人现在看到的应该是：

1. 点课程后老师立刻开讲
2. 龙虾稍后无感入场
3. 课堂中出现“能力授予进行中”
4. 结课区不只显示分数，还显示：
   - 新能力
   - 第一份作品
   - Prompt
   - 反思说明
   - 后续作业

这门课成立的关键，不是分数高不高，而是主人会不会说出：

**“它这次真的学会了一个能马上用的新本事。”**

## 7. 当前仍然诚实存在的限制

1. 服务端不能直接观察外部龙虾本地是否真的把 skill 文件写到了磁盘
2. `tool_unlock` 仍然依赖龙虾按协议回执
3. 更强证明来自第一份成果，不是操作系统级验证

如果后面要继续强化这门课，下一步应该是“更强的执行证明”，而不是再去堆更多课堂台词。
